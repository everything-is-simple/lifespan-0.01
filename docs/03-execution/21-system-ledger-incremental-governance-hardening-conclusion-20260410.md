# 全系统历史账本增量治理硬约束结论

结论编号：`21`
日期：`2026-04-10`
状态：`生效`

## 裁决

- 接受：全系统正式账本统一遵守“稳定自然键 + 一次性批量建仓 + 后续增量更新 + 断点续跑 + 审计账本”的历史账本机制。
- 接受：标的类正式主锚默认采用 `asset_type + code`；`name` 只作属性、快照或审计辅助字段。
- 接受：所有后续正式卡必须显式填写 `历史账本约束` 六条声明，否则不得进入正式实现。
- 接受：`check_doc_first_gating_governance.py` 已升级为这条规则的正式硬门禁。
- 拒绝：把 `code + name` 冻结成全系统唯一主键。
- 拒绝：继续容忍“按 run_id 说话”的一次性流水线写法进入正式账本。

## 原因

- 卡 `17/19/20` 已经在 `raw/base` 证明：只有把建仓、增量、续跑、审计都做成正式机制，系统才符合“长期可运行、可续跑、可复算、可审计”的目标。
- `name` 会改名，不能承担全系统稳定主锚职责；正式主锚必须先落在稳定实体锚点上。
- 如果不把这套机制提升成门禁，下游模块仍可能各写各的 checkpoint、自然键与 replay 语义，重构目标会再次漂移。

## 影响

- 最新正式生效结论锚点切换到卡 `21`。
- 后续任何正式卡在进入 `src/`、`scripts/`、`.codex/` 前，都必须显式声明：
  - 实体锚点
  - 业务自然键
  - 批量建仓
  - 增量更新
  - 断点续跑
  - 审计账本
- 卡 `20` 的 `stock/index/block raw->base` 机制现已不只是 data 模块经验，而是全系统正式治理标准。
