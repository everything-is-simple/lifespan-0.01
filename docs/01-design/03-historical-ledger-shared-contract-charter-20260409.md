# 历史账本共享契约设计

日期：`2026-04-09`
状态：`生效中`

## 背景

新仓已经冻结了多库结构、五根目录和模块边界，但“历史账本”目前仍主要停留在原则层。
如果不把共享契约写死，后续 `position / alpha / portfolio_plan / trade` 很容易各写各的自然键、审计字段和增量语义。

## 设计目标

1. 把“数据库不是一次运行产物，而是历史账本”落成可执行契约。
2. 冻结自然键、审计字段、增量更新、断点续跑和写入边界的共识。
3. 明确模块名、账本物理名和五根目录来源的共享口径。
4. 让后续模块实现先服从共享账本契约，再谈各自业务细节。

## 设计原则

### 1. 自然键优先

所有正式账本以业务自然键为主语义，不以 `run_id` 充当主键。
实体锚点默认由 `code + name` 组成，再叠加：

1. 业务时间键
2. 窗口键
3. 策略键
4. 场景键
5. 状态键

### 2. 审计字段与业务字段分离

`run_id`、构建批次、写入时间、来源摘要、版本标签等只属于审计层。
它们必须保留，但不能篡夺业务自然键。

### 3. 账本与快照分层

1. `ledger / fact / event` 表默认追加写入
2. `snapshot / latest_state` 表允许按自然键重算覆盖
3. 快照必须可追溯到对应账本或事实层，不允许成为不可回溯黑箱

### 4. 正式环境与测试环境分层

正式运行时必须优先使用环境变量或 `scripts/setup/enter_repo.ps1` 固定五根目录。
`repo_root.parent` 形式的相邻目录推导只作为测试和开发兜底。

## 共享影响范围

本设计影响以下正式账本：

1. `raw_market`
2. `market_base`
3. `malf`
4. `structure`
5. `filter`
6. `alpha`
7. `position`
8. `portfolio_plan`
9. `trade_runtime`
10. `system`

## 关键命名映射

1. 模块边界名是 `trade`
2. 物理账本名是 `trade_runtime.duckdb`
3. 文档与代码都必须承认这是一组明确映射，而不是两个含糊同义词

## 不在本设计内

1. 各模块具体业务表结构
2. 各策略的独立算法逻辑
3. `position` 资金管理表族的细节拆分
