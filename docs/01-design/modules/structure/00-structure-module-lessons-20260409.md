# structure 模块经验冻结

日期：`2026-04-09`
状态：`生效中`

## 当前职责

- 从 `malf` 语义层中外提稳定、可复用的价格结构事实
- 统一回答“发生了什么结构事实”
- 为 `filter / alpha` 提供独立结构入口，而不是继续直连旧现象链

## 必守边界

1. `structure` 只整理结构事实，不负责准入裁决。
2. 下游应优先消费 `structure_snapshot` 一类结构出口，而不是继续咬住 `phenomenon` 内部实现。
3. `structure` 是正式模块，应该沉淀为可复查、可增量、可冻结的结构账本。

## 已验证坑点

1. 结构事实长期塞在 `malf` 现象链里，会让下游语义耦合越来越重。
2. 没有稳定结构出口时，`filter / alpha` 很容易重复解释同一件结构事实。
3. 把结构判断和 trigger 判定混在一起，会让后续账本根本接不上主线。

## 新系统施工前提

1. 先冻结结构事实词汇与自然键，再谈更复杂的结构研究。
2. 结构事件与结构快照要分层，便于增量更新和断点续跑。
3. 后续所有结构扩展都不得越界替代 `filter` 或 `alpha` 的职责。

## 来源

1. 老系统 `malf 30` structure 事实层外提章程
2. 老系统总表中对结构语义与下游桥接的总结
